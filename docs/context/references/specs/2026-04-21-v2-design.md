# Squid-Tools v2 Design Spec

## Purpose

v1 ships the core connector: a continuous-zoom viewer, selection + live processing, 5+ algorithm plugins, R2 hosting, 3D rendering primitives, and an Algorithm Absorber that turns an external repo into a plugin in one agent pass.

v2 takes the loose ends from the master v1 spec's "OUT (Future Cycles)" list that were explicitly **not** closed in v1, plus the second-order items that fell out of v1 testing feedback. It is the "scale and polish" release.

**Audience:** Life-sciences users with larger datasets, distribution at scale, and agents that can absorb algorithms without human review.

**Guiding principle:** v1 is the skeleton; v2 is the muscle. Every v2 cycle should make an existing user-facing surface feel smoother or open a new surface that v1 punted on.

---

## v2 Scope

### A. Distribution & packaging

**A1. Web download page for modular bundles**
A Cloudflare Pages site where a user checks boxes for the plugins they want (stitching, flatfield, decon, phase, aCNS, bg-subtract, segmentation, tracking, …), clicks Download, and gets a `.exe` or `.AppImage` bundling exactly those modules. CI watches the `processing/` directory; any new plugin triggers a rebuild of the per-module bundles.

**A2. Checkbox installer UI**
A small desktop installer that presents the same module list and installs per-module into a sandboxed location. Runs the post-install smoke test for each chosen module before reporting success.

**A3. PyInstaller hardening**
Current PyInstaller spec covers core. v2 bundles plugin-specific deps (cupy-cuda12x for GPU compositing, numba for stitching, BaSiCPy for flatfield) into their respective module bundles. Resolves "user pip-installs and something breaks" on Windows.

### B. Algorithm coverage (second-wave absorptions)

**B1. Cell segmentation** — StarDist, CellPose, or a lightweight in-house U-Net. Per-tile inference. Live behavior: paint overlays as each tile completes. Includes the "magical brush" annotation tool (user paints rough, model refines).

**B2. Cell tracking** — transfer-learned tracker over a time-series. Live behavior: user clicks a cell in the segmentation overlay, track extends forward + backward across timepoints.

**B3. Object detection** — generic bounding-box detector plug-in (YOLO-style). Per-tile, live box overlay.

**B4. Smart Acquisition (CRUK 3D)** — closed-loop acquisition control. Reads a real-time feed from the microscope, runs an inference model (e.g. a segmentation network), and sends commands back (stage moves, focus adjustments, new z-plane) to re-acquire the regions of interest. This is the "hot loop" for acquisition automation.

**B5. Media Cybernetics metadata interop** — reverse-engineer / absorb their metadata conventions so a Media-Cybernetics-captured dataset opens cleanly in squid-tools. Pair with a reader auto-generator (C1).

### C. Data & format

**C1. Auto-generated reader classes** — a generator that reads a format spec (YAML / prose) and produces a `FormatReader` subclass. Goal: adding a new Squid output format becomes one YAML file, not a bespoke reader.

**C2. Formal OME schema validation** — the sidecar manifest and any OME-TIFF export get validated against the published OME-XML XSD. Lint pass on ingestion. Makes squid-tools interoperable with Bio-Formats.

**C3. Petabyte-scale optimizations** — the v1 primitives (LRU cache, handle pool, spatial index, pyramid) work for tens of GB but haven't been benchmarked at TB+. v2 adds:
- Disk-backed pyramid cache (v1's is in-memory only)
- R-tree spatial index (v1 uses a uniform grid; fine for regular mosaics, weaker for irregular scans)
- Tile prefetching (speculatively load tiles one ring out from the viewport)
- Zero-copy dask slab reads for OME-TIFF
- Benchmark suite with synthetic 10 TB dataset + latency targets

### D. User surface

**D1. Navigator pane** — persistent mini-map in a corner showing the full acquisition with a highlighted viewport rectangle. Reuses the pyramid (highest level = navigator). Click-to-jump.

**D2. Annotation tools** — polygon / brush / freehand drawing on the stage view. Output: `.squid-tools/annotations.geojson`. Feed into segmentation (B1) for weakly-supervised training.

**D3. Viewer preferences panel** — default colormaps, default clims, default pyramid aggressiveness, FOV border style. Persist per-user.

**D4. Acquisition comparison** — side-by-side split view for two acquisitions (same sample, different timepoints; before/after a perturbation). Sliders synchronize.

### E. Programmatic surface

**E1. MCP (Model Context Protocol) API server** — expose the full plugin + viewer API via an MCP server so Claude or any other LLM harness can drive squid-tools programmatically. Endpoints: `load_acquisition`, `list_plugins`, `run_plugin`, `get_frame`, `export`. High-throughput scripting for pipelines.

**E2. Scriptable pipelines** — a YAML/JSON pipeline format that chains plugins in sequence with param overrides. Run via CLI, MCP, or a "Pipeline" tab in the GUI.

### F. Distribution + Demo Network (from v1 spec, deferred)

**F1. Cloud CI + R2 bundle publishing** — on a commit to `processing/*/plugin.py`, GitHub Actions builds the per-module bundles and uploads them to the R2 bucket that the download page serves.

**F2. Browser viewer improvements** — v1 lands a minimal HTML/Canvas2D page streaming from R2. v2 upgrades to WebGL (ideally via vispy.jsx or a lightweight alternative), adds pan/zoom, multi-channel composite client-side, and selection/run over remote data.

**F3. Three-node acquisition → absorb → demo loop** — the full network from the v1 spec's "Development + Demo Network" section: dev machine runs the absorber, pushes to git, CI rebuilds bundles, web demo auto-updates. v1 has the pieces; v2 wires them together with Actions + a `status.json` on R2 the download page polls.

### G. Testing + CI

**G1. Full-product functional testing plan** — per the feedback memory on spec conventions, the testing plan is written when v1 is complete. Covers: open every supported format, run every plugin on a canonical dataset, verify outputs match golden files, performance regression on the petabyte benchmark.

**G2. Agentic regression runner** — an agent that, on every PR, pulls the PR's changes, runs the functional test plan, and posts a review. Closes the "agents add modules, humans don't review every line" loop.

### H. Future-looking (stretch)

**H1. Real-time acquisition streaming** — not just post-processing. Open squid-tools while an acquisition is running and see tiles arrive live. Required groundwork for Smart Acquisition (B4).

**H2. WebGL VolumeVisual port** — 3D rendering in the browser (v1 3D is desktop-only via vispy + OpenGL).

**H3. Collaborative viewing** — two users open the same R2-hosted acquisition, share cursor + selection + run commands. Over WebSockets.

## Sequencing

A plausible v2 release order (not a contract):

1. G1 — write the functional testing plan first (grounds everything else)
2. A1 + F1 — web download page + cloud CI (needed to distribute every subsequent cycle)
3. A2 + A3 — installer hardening
4. C3 — petabyte-scale optimizations (unblocks large-customer demos)
5. F2 — browser viewer WebGL
6. B1 — cell segmentation (most-requested algorithm after stitching + flatfield)
7. D1 — Navigator pane (low-cost quality-of-life)
8. B2 + D2 — tracking + annotation tools (complementary)
9. E1 — MCP API
10. Remaining: B3, B5, C1, C2, D3, D4, E2, F3, G2
11. Stretch: B4, H1, H2, H3
